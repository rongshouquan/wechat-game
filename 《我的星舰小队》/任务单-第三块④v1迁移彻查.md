## 背景（为什么查）

2026-07-06 一天内在 S10.5 连抓两处同类漏迁（3天行动/7天扩张的"过程/完成/结算"三层奖励在生成 v2.0 时被压缩丢失，Ron 指认后已复原）——系统性丢失实锤。本单把 v1.0 全文扫一遍，找出所有同类问题。已复原的两处可当"漏迁长什么样"的参照样本（见 GDD-v2.0 S10.5 标注）。

## 方法（硬规格）

1. **逐节逐条扫 v1.0 全文**，每条在 v2.0 体系里找归宿，内部判定四态：已迁一致 / 有拍板依据的改写 / **漏迁** / 有意废弃（如 v1.0 里已标"已砍"、或 README 锁定决策明确取代）。
2. **只交付红名单** = ①漏迁 ＋ ②"改写了但查不到拍板依据"的存疑项。绿的（一致/有据/有意废弃）不写入交付。
3. 红名单每条格式：v1.0 原文引用（带节号）→ v2.0 现状（缺失/被改成什么）→ 影响面一句（涉及哪个系统/数值）→ 处置建议一句（复原原文 / 补拍板 / 确认废弃）。**只审不改**：零代码、不动 GDD、不动配置。
4. **防漏扫自证**：完成报告里给覆盖计数（v1.0 共扫出 N 个条款、判定四态各多少条），并列出你**跳过或无法判定**的部分——宁可标存疑，不许静默略过。
5. 判定基准：v2.0=现行真源，正常冲突以 v2.0 为准；红名单只抓"**丢了**"和"**改了但没人拍过板**"两类。拿不准一律进红名单标〔存疑〕，由总控/Ron 裁。
