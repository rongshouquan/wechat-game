# 第二块 · 关卡战斗内容生产规范（2c · 给 Codex 批量走量用）

> 背景：150关拓扑（6星域/25章节/6Boss）已经落地（见 `game/tools/generate-s7-mainline-topology.mjs`），
> `mainline_node_config` 150行的 `templateRef`/`problemTagRef`/`nodeTypeTag` 都已经定好，**不用再设计"这关是什么主题"**，
> 这一步要做的是：把每一关"敌人具体怎么摆、打多久、给多少压力值"的战斗数据填出来。
> 现状：150关里只有 9 关（n001-n007 + n084 + n150）有完整战斗数据，其余 141 关都是空的，交给 Codex 批量产出。
> 产出后跑 `npm run validate:configs:s7`（在 `game/` 目录下），必须显示 `all s7 configs valid` 才算过关。

## 1. 要填哪几张表

在 `game/assets/resources/configs/s7/` 目录下，这一步涉及 4 张表：

| 表名 | 作用 | 现状 |
|---|---|---|
| `battle_unit_stat_param.sample.json` | 敌人单位的属性（血/攻/防/攻击间隔等） | 已有8个通用敌人(enemy_swarm/shield/backline/support/burst/charge/summon/boss_add) + 2个真实Boss(bu_boss_n084/bu_boss_n150)，够用；新增内容主要是"给新Boss阶段召唤的附属单位"如果需要的话 |
| `battle_encounter_param.sample.json` | 某一关"用哪个模板+哪些敌人+压力值多少" | 141行待补，1行=1关 |
| `battle_spawn_param.sample.json` | 某一关"敌人具体摆在哪个格子、第几波刷出来" | 跟着 encounter 走，1关通常1-2行 |
| `battle_boss_phase_param.sample.json` | Boss关"第几阶段触发什么" | 4个Boss(n060/n102/n120/n138)待补，每个Boss通常3行(start/mid/final) |

## 2. 每一关怎么填——照着 mainline_node_config 里已经定好的字段来，不用自己判断

对 n008 到 n150 之间**除 n018/n019（保护期转折，非战斗节点，跳过）**外的每一个节点：

1. 打开 `mainline_node_config.sample.json`，找到该节点的 `templateRef`、`problemTagRef`、`nodeTypeTag`、`starfieldId`。
2. 按 `nodeTypeTag` 决定 `stageType`：`normal`→`normal`，`elite`→`elite`，`boss`→`boss`（跟 mainline 的 tag 保持一致，这个字段校验器会查）。
3. `pressureRef` 按 `starfieldId` 找对应的 `np_sf0X`（普通关）或 `ep_sf0X`（精英关）行（`pressure_param.sample.json` 里已经有全部6个星域的区间了）；Boss关用对应的 `bp_nXXX` 行（6个都已经建好占位区间，见下表）。
4. `enemyUnitStatRefs`：按 `problemTagRef` 从下面"敌人对照表"里选1-2个通用敌人单位；同一关可以混用主标签+复用标签（`secondaryPressureTag`不是none时）对应的敌人，制造变化。
5. `spawnPlanRefs`：给这个encounter配1-2波刷怪（`spawnPlanRefs`引用的行要新建在`battle_spawn_param`里，`waveIndex`从1开始），格子用`slotRefs`（比如`r0c0`），别超出5行7列的战场范围。
6. `timeLimitSec` 统一 120（跟现有关卡一致，不用改）。
7. `battleSeedPolicy` 统一 `node_id_plus_run_seed`。

## 3. 4个待补Boss的机制怎么落成配置——照抄GDD附录D §5的文字描述拆成阶段

每个Boss需要：1行`battle_unit_stat_param`(boss本体属性，占多格用`sizeRows`/`sizeCols`) + 1-2行召唤物用的普通敌人属性(如果有召唤机制，复用上面"敌人对照表"里已有的) + 1行`battle_encounter_param` + 2行`battle_spawn_param`(boss本体+附属) + 3行`battle_boss_phase_param`(start/mid/final，`triggerType`用`battle_start`或`hp_pct_below`)。

**照抄 n084（失控母舰）或 n150（星能污染核心）现有4张表里的写法当模板**——字段结构、数值量级参考着来，具体数值（血量/攻击/召唤上限）都是占位，不用精算，第二块数值校准阶段会统一重调。

## 4. 硬性红线（校验器会挡，别踩）

- `召唤上限summonCountCap` 不能超过10（`battle_boss_phase_param`）。
- Boss（`stageType=boss`）用的 `templateRef` 必须是该节点在 `boss_node_config` 里登记的那个（n060用t02、n084用t04、n102用t06、n120用t09、n138用t08、n150用t10，已经定好了，别改）。
- 只有 n150 允许用 t10。
- 每个 encounter 的 `problemTagRef`/`templateRef`/`stageType` 必须跟 `mainline_node_config` 对应节点完全一致（这是最容易出错的地方，两边要对得上）。
- 不要发明新的攻击弹种/新机制——所有敌人机制必须能在GDD附录D的敌人清单里找到对应（数量/护盾/召唤/点名/续航/削弱/狂暴 这几种之一）。
