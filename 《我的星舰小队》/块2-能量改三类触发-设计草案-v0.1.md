# 块2 设计草案：能量条 → 技能三类触发（v0.1，待 Ron 拍板）

> 目标：把 S7 引擎现在的"攒能量满了放大招"整套，换成 v1.0 §4.2 的"取消能量条 + 三类异构触发"。
> 这是 8 块里**最大的架构改动**，会动战斗主循环、几乎所有战斗测试基线都会变。本草案先定打法、列风险、标决策点；**不含代码**。

---

## 1. 现状机制（精确，引用 S7AutoBattleEngine.ts）

- 每 tick 9 步（L182-195）：① 出怪 ② 状态到期 ③ **被动回能** ④ **满能放大招** ⑤ 普攻(顺带涨能) ⑥ **普攻后满能放大招** ⑦ Boss阶段 ⑧ 清死亡 ⑨ 判胜负。
- 能量：`ENERGY_FULL=100`；`stepPassiveEnergy` 每 tick `energy += passiveEnergyPerSec * 0.2`（L335-341）；普攻命中 `+ENERGY_ON_ATTACK=12`（L368）；受击 `+ENERGY_ON_HIT=4`（约 L567）。
- 放大招 `tryCastUltimate`（L418-439）：`energy>=100` 且未被控制（短路/晕眩）→ 放 `ultimateEffectRef`，`energy-=100`；**首次大招后触发一次 `coreEffectRef`**（`coreTriggered` 标志）。
- 普攻 `normalEffectRef`：每 `attackIntervalSec` 一次，与能量无关（L350-373）。
- **注意**：因攻击/受击涨能，活跃单位实际约每 5-8 秒放一次大招（远快于纯被动的 ~17 秒）——这点在改造后会变。

## 2. 目标机制（v1.0 §4.2 + 决策日志 §137）

- **取消能量条**。
- 技能**三类异构触发**（靠 effect 多样性天然错峰，不靠统一 CD 硬错峰）：
  - **被动型**：常驻（强化普攻 / 光环）。
  - **条件型**：满足条件触发（击杀 N 个 / 友军阵亡到数 / 血量 < X%）。
  - **CD 型**：每隔 CD 自动释放，**默认开局即放第一次**；区间约 10-30s。

## 3. 关键利好：块1 已备好触发数据模型

块1 的 `S7TriggerBlock`（`S7BattleEffectBlock.ts`）已定义：
`{ on: 'battle_start'|'cd'|'on_kill'|'hp_below'|'on_hit'|'passive', cdSec?, threshold?, effectRef }`，
装配层 `deriveUnit` 已把它们收集进 `derived.triggers`。所以块2 = **让引擎消费 triggers**，数据结构现成。

## 4. 映射方案（现状概念 → 三类触发）

| 现状 | 块2 后 |
|---|---|
| 普攻 `normalEffectRef`（每 interval） | **不变**（普攻是"持续输出"，不是三类触发之一） |
| 被动型（强化普攻/光环） | 走**块1 的 modifier/affix 积木**（装配时叠加），**不在 tick 里 fire** |
| 大招 `ultimateEffectRef`（能量满放） | 变成一条 **CD 型触发**：`{on:'cd', cdSec, effectRef: ultimateEffectRef}` |
| 条件技（血量低/击杀后…） | **条件型触发**：`hp_below` / `on_kill` / 友军阵亡数 |
| `coreEffectRef`（首次大招后） | 暂保留"首次触发任意技能后放一次"的钩子（完整星核留**块3**） |
| `passiveEnergyPerSec` 字段 | 复用为"大招 CD 来源"：`cdSec = 100 / passiveEnergyPerSec`（见决策点①） |

**所有单位（玩家+敌人）共用引擎**，故 CD 触发对两边统一生效。敌人不走装配层 → 由引擎按其 `ultimateEffectRef + passiveEnergyPerSec` **自动派生**默认 CD 触发；玩家在此基础上由装配层叠加/覆盖触发（块3/4/5 内容）。

## 5. 新 tick 循环（块2 后）

① 出怪 ② 状态到期 ③ ~~被动回能~~→**评估触发(stepTriggers)** ④ ~~满能放大招~~（并入③）⑤ 普攻(去掉涨能) ⑥ ~~普攻后满能放大招~~（并入③/事件钩子）⑦ Boss阶段(不变) ⑧ 清死亡 ⑨ 判胜负。

`stepTriggers`：遍历每个存活、未被控制的单位的 triggers：
- `battle_start`：t=0 放一次。
- `cd`：`time >= nextFireAt` → 放，`nextFireAt += cdSec`；**首次 nextFireAt=0（开局即放，见决策点②）**。
- `hp_below`：`hp/maxHp < threshold` 且本触发未放过 → 放（一次性闩锁）。
- `on_kill` / `on_hit`：事件驱动（击杀/受击时置标志，下次 stepTriggers 评估）——**留块2b**。
- `passive`：不 fire（已在装配层当 modifier）。
- 控制（短路/晕眩 `hasControl`）：仍拦截，被控时不放。

## 6. 迁移分步（避免一次打挂全部测试）

- **块2a（核心替换，最伤测试基线）**：
  - `RtUnit` 加 `triggers` + 每条的 `nextFireAt`/`fired` 状态；spawnUnit 时：玩家用 `derived.triggers`，所有单位再**自动补一条默认 CD 触发**（由 `ultimateEffectRef`(≠none) + `cdSec=100/passiveEnergyPerSec` 派生）。
  - 删除能量三步（被动回能/两处满能放大招）+ **所有 `gainEnergy` 调用点**（普攻涨能 L368 + 受击涨能约 L567）+ `energy` 字段 + `energy_change` 日志。
  - **（自检补）`finalState` 输出含 `energy` 字段（buildFinalState L881）+ 其类型 `S7AutoBattleUnitFinalState`**：移除能量后要一并删字段+改类型+改断言它的测试（否则编译/测试挂）。
  - **（自检补）自动 CD 派生的边界**：仅对 `ultimateEffectRef≠'none'` **且 `passiveEnergyPerSec>0`** 的单位派生（防 100/0=∞）；rate≤0 或 none → 该单位只普攻、无 CD 触发。
  - **（自检补）短路语义**：现状短路只在 `tryCastUltimate` 拦大招；改后短路应拦**所有触发 fire**（对齐 v1.0"无法触发技能"），实现时确认 `canAct`（管普攻）与 `hasControl`（管技能）边界不串。
  - 新增 `stepTriggers` 处理 `battle_start`/`cd`/`hp_below`。
  - `coreEffectRef` 钩子改为"首次任意触发 fire 后放一次"。
  - **重定基**受影响的战斗测试（durationSec/outcome/能量相关断言）。
- **块2b（条件型补全）**：`on_kill`/`on_hit`/友军阵亡数 事件钩子；装配层条件型积木接入；被动型确认全走块1 modifier。

## 7. 关键行为变化 & 风险

1. **大招频率大变 + 测试重定基（头号风险）**：现状活跃单位 ~5-8s 一发（攻击涨能），改 CD 后 ~17-25s → 大招明显变稀。**（自检反而是好事）**：现状 ~5-8s 其实**比 v1.0 想要的 10-30s 还快**，改 CD 后落进 v1.0 区间，是更对齐、不是退化。但代价是几乎所有 `s7_auto_battle_engine.test.ts`（~31 用例）的 durationSec/outcome/能量断言都要变。
   - **执行纪律（重要）**：重定基不能"为了让测试绿就改数字"——每个变化的期望值都要**按新机制核实它为什么对**（不制造虚假验收）。这是块2a 的最大执行风险，必要时拆更细或逐个说明。
   - CD 具体值是数值：块2 只立机制、给占位默认（决策点①），最终由第二块调。另可选：把占位 CD 调到接近现状 ~6-8s 以减少重定基量，但那偏离 v1.0 的 10-30s——我倾向按 v1.0 立机制、接受重定基。
2. **开局齐放**：所有带大招的单位 t=0 齐放第一发（v1.0 认可，靠 effect 多样性而非错峰）。原型期占位 effect 下可能显得"开局炸一片"。（决策点②）
3. **`energy_change` 日志移除** → 断言它的测试要改。
4. **重复触发风险（其实是块3+ 才需处理）**：块2a 本身干净——每个单位恰好一条自动 CD 触发，无重复。只有当块3/4/5 给玩家加触发、可能重新指定大招时才需去重规则。届时规则：星舰自带大招恒为默认 CD 触发；驾驶员/插件/星核只**新增**触发，除非某积木显式声明"覆盖大招槽"。块2a 不必实现去重。
5. Boss 阶段（step7）独立、不动；但其释放走同一 cast 路径，验证仍有效。

## 8. 需 Ron 拍板的决策点

1. **CD 来源**：(A) 复用 `passiveEnergyPerSec` 反推 `cdSec=100/rate`（零配置改动，落点 ~17-25s 在 v1.0 的 10-30s 内）；(B) 新增显式 `ultimateCdSec` 字段到 battle_unit_stat_param（更清晰，但要动配置表+校验+.mjs+测试）。**我推荐 A**（先零改动立机制，后续要精细再加显式字段覆盖；具体值留第二块）。
2. **开局即放**：按 v1.0"CD 型开局即放第一次"→ 开局齐放第一发。确认；还是想要"首发延迟到 cdSec 后"（更平滑但违 v1.0 字面）？
3. **`coreEffectRef`**：块2 暂保留"首次触发后放一次"的钩子，完整星核质变留块3。确认。
